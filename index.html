OAKLEON

Version control for information topologies

Evolving Information

Oakleon is creating a new kind of data architecture. Our system adapts at runtime as information, behaviors, and new models arise. Put another way, the program changes shape to fit the data instead of forcing the data to fit the program. Any kind of record can be forked, merged, committed, and rolled back, like distributed version control for structured data. This opens up all kinds of possibilities in areas like business operations, resource management, and collaborative investigation. Oakleon is looking for talented developers to push this technology forward.

An open problem

Today there are thousands of business applications for managing information. Everything from customer lists and legal forms to experimental research, each with a slightly different way of doing the most basic tasks. Yet each of these programs do many of the same things, present forms for users to fill in, provide query operations, and display results in a report. Steadily these programs grow in complexity and scope until there is significant overlap in functionality. This leads to problems with training users on many programs, double entry, and even a circular dependency of systems. A generalized approach to gathering, storing, and reporting structured information is needed.

Solutions

Project Oakfield

A system of systems designed to solve the problem of storing heterogeneous structured records with near real-time user updates. Oakfield should overcome the need to describe a schema before data curation and entry thereby allowing the users to iterate on the structure over time. This system uses modern open source technologies as a foundation so individual components can be updated and modified as new systems develop without lock-in to a proprietary vendor. Oakfields core components include a distributed cache, graph database, full-text search, Isotope (our custom protocol), and worker cluster.

Oakfield API example

// create a record
r1 = {
    id:         1,
    dob:        "3/14/1592",
    type:       "user",
    name:       "john doe",
    tags:       ["tech", "manager"]
    number:     "196-9665",
    namespace:  "acme001"
}
// commit the record
Oakfield.commit([r1])

// Link heterogeneous records together

// some equipment data
e1 = {..., type:"vehicle" }

// some sensor data
s1 = {..., geo:[32.80, -117.17]}

// Link the records with tags
Oakfield.link(r1, e1, ["uses"]);
Oakfield.link(e1, s1, ["tracks"]);
Oakfield.commit([r1,e1,s1]);

Project Canopy

An adaptable user interface. Every element updates dynamically as records and topologies change. Canopy automates UI generation by generalizing common use patterns for many kinds of applications. No building input forms, boilerplate endpoints, or synchronizing state. This UI builds itself from the data it is asked to represent.

Canopy API example

// create list UI structure with a query
v1 = {
    viewtype:   "list",
    querytype:  "graph",
    p1:         "customer",
    p2:         "service",
    p3:         timespan(30,"days"),
    p4:         now - p3;
    q: (p1) -> (p2 .date < p4)
}
// push the view onto the stack
