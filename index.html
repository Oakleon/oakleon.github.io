Evolving Information

Data management is broken. The typical rigid structures for storing and reporting data are ineffective and inefficient. Data management should be built around the data, and not the other way around.

Because markets are constantly shifting and evolving, the same is true for the data that represents them. The solution is simple: make the software adapt to fit the data, even as new types of data become part of the picture. Oakleon is looking for talented developers to push this technology forward.

Solutions

Project Oakfield

Our adaptable data management system is designed to solve the problem of storing heterogeneous structured records. Users iterate on the structure of the data over time, making Oakfield a moment-to-moment solution that rises to demand. Featuring near real-time user updates, and no need for schema descriptions before entry or curation, Oakfield solves the pervasive problems in data management.

Oakfield API example

// create a record
r1 = {
    id:         1,
    dob:        "3/14/1592",
    type:       "user",
    name:       "Leonhard Euler",
    tags:       ["tech", "manager"]
    number:     "196-9665",
    namespace:  "acme001"
}
// commit the record
Oakfield.commit([r1])

// Link heterogeneous records together

// some equipment data
e1 = {..., type:"vehicle" }

// some sensor data
s1 = {..., geo:[32.80, -117.17]}

// Link the records with tags
Oakfield.link(r1, e1, ["uses"]);
Oakfield.link(e1, s1, ["tracks"]);
Oakfield.commit([r1,e1,s1]);

// Modify
r1 = Oakfiled.query.id(1);
f1 = {status:"active"};
Oakfield.append(f1, r1);

// add a field to many records
records = Oakfiled.query.tag("user");
f1 = {status:"active"};
Oakfield.append(f1, records);

// Query
Oakfiled.query.tag("user");
Oakfiled.query.geo(33.02, -117.41, 5);
Oakfiled.query.graph("(:user)->(:friends)");
Oakfiled.query.fulltext("euler");

Project Canopy

Our data management solution is adaptable, and the interface for that data should be, too. Based on common use patterns, Canopy suits the needs of the task. Because the interface adapts to demand, there’s no building input forms or boilerplate endpoints.

Canopy API example

// create list UI structure with a query
v1 = {
    viewtype:   "list",
    querytype:  "graph",
    p1:         "customer",
    p2:         "service",
    p3:         timespan(30,"days"),
    p4:         now - p3;
    q: (p1) -> (p2 .date < p4)
}
// push the view onto the stack
Canopy.viewstacks[0].push(v1);

// the UI is updated automatically
// and the view is kept up to date

// add a field
r1 = Oakfiled.query.id("14794b5df8f");
f1 = {status:"active"};
Oakfield.append(f1, r1);

// add a field to many records
records = Oakfiled.query.tag("user");
f1 = {status:"active"};
Oakfield.append(f1, records);

// every client viewing these records 
// will be updated

// timeline
v1 = {
    viewtype:   "timeline",
    querytype:  "tag",
    q:          "events"
}

// push the view onto the stack
Canopy.viewstacks[0].push(v1);

// the UI is updated automatically
// and the view is kept up to date

// map
v1 = {
    viewtype:   "map",
    querytype:  "geobox",
    p1:         [33.0345,-117.8234],
    p2:         [33.1575,-117.6004],
    q:          p1,p2
}
// push the view onto the stack
Canopy.viewstacks[0].push(v1);

// the UI is updated automatically
// and the view is kept up to date
